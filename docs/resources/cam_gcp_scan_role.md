# visionone_cam_gcp_scan_role (Resource)

Trend Micro Vision One Cloud Account Management GCP read-only scan role. Creates a custom GCP IAM role that holds only read permissions (resource hierarchy discovery plus Cloud Asset Inventory read) for GCP auto-detection. Grant it once at the organization or folder node via `node_scan_roles` on `visionone_cam_service_account_integration`; projects under the node — including projects created later — inherit it through IAM. Unlike `visionone_cam_iam_custom_role`, it never includes deploy/write permissions, so enabling a feature only appends that feature's read permissions.

## When to Use This Resource

Create this resource **only when you enable GCP auto-detection**. It defines the read-only role that lets Vision One discover and scan projects across an organization or folder. Bind it once at the Org/Folder node via `node_scan_roles` on `visionone_cam_service_account_integration`; every project under the node — including projects created later — inherits it through GCP IAM, so no per-project scan binding is required.

For the deploy/self-maintenance role bound to the primary project, use `visionone_cam_iam_custom_role` instead.

## Granted Permissions

The role is created with a read-only permission set and never includes deploy/write permissions:

- Resource hierarchy discovery (equivalent to `roles/browser`): `resourcemanager.folders.get`, `resourcemanager.folders.list`, `resourcemanager.organizations.get`, `resourcemanager.projects.get`, `resourcemanager.projects.getIamPolicy`, `resourcemanager.projects.list`
- Cloud Asset Inventory read (equivalent to `roles/cloudasset.viewer`): `cloudasset.assets.searchAllResources`, `cloudasset.assets.searchAllIamPolicies`, `cloudasset.assets.listResource`, `cloudasset.assets.exportResource`, `cloudasset.feeds.get`, `cloudasset.feeds.list`

Adding `feature_permissions` appends only that feature's **read** permissions. `roles/viewer` is a basic role and cannot be inlined into a custom role; grant it separately at the node (via `node_scan_roles`) when viewer-level read is needed.

**Folder caveat:** GCP custom roles have no folder scope. For folder deployments the scan role is still DEFINED at the organization level (set `organization_id`, which requires organization-level permission) and BOUND at the folder node.

## Example Usage

```terraform
# Example: GCP read-only scan role for auto-detection
#
# Create this role only when you enable GCP auto-detection. It holds read-only permissions
# (resource hierarchy discovery + Cloud Asset Inventory read) and is granted once at the
# organization or folder node via node_scan_roles on visionone_cam_service_account_integration,
# so every project under the node — including projects created later — is covered through IAM
# inheritance. Unlike visionone_cam_iam_custom_role, it never includes deploy/write permissions.

terraform {
  required_providers {
    visionone = {
      source = "trendmicro/vision-one"
    }
  }
}

provider "visionone" {
  api_key       = "your-vision-one-api-key"
  regional_fqdn = "https://api.xdr.trendmicro.com"
}

# Custom roles have no folder scope, so the scan role is DEFINED at the organization level
# (organization_id) and BOUND at the folder/organization node. Defining an org-level role
# requires organization-level permission.
resource "visionone_cam_gcp_scan_role" "scan_role" {
  project_id      = "my-management-project" # used for GCP authentication
  organization_id = "123456789012"
  role_id         = "trend_ai_auto_detect"
  title           = "Trend Vision One Auto-Detect Scan Role"
  description     = "Read-only discovery and scanning role bound at the organization or folder node"
}

output "scan_role_name" {
  value       = visionone_cam_gcp_scan_role.scan_role.name
  description = "Full resource name of the scan role; pass it to node_scan_roles on visionone_cam_service_account_integration"
}
```
